# v0.2 redesign — Scenarios over Capabilities

> Draft, 2026-06-15. Proposal for v0.2.0. Discussion document — no code
> yet. Once we agree on the model, we cut a feature branch and
> implement.

## TL;DR

The v0.1.x canvas exposes four orthogonal flags (`AllowParameters`,
`AllowComponents`, `AllowScripting`, `ComponentScope`+`CategoryFilter`).
That's an engineer-facing surface: 16 boolean combinations of which
~5 are meaningful, and the user has to do the meaning lookup in
their head every time.

v0.2 replaces that surface with a single **Scenario** dropdown —
five named modes that map onto the underlying capability flags. The
mapping is hidden by default; an "Advanced" expansion still exposes
the raw flags for the few users who need them.

Same engine, smaller and more honest UI. The conceptual work goes
into two new modes that need real infrastructure: **Coach** (turn-
bounded change tracking + visual highlighting) and **Execute**
(Skill-defined command grammar, AI cannot stray).

In parallel the Skills schema goes from "free-form markdown" to
versioned `SKILL.md` frontmatter + structured directories
(`components/`, `reference/`, `images/`, `prompts/`, `tests/`).

## Why the current design is hard to use

Today, dropping the MCP Server component on the canvas exposes 8
inputs: `Run`, `AllowParameters`, `AllowComponents`, `AllowScripting`,
`ComponentScope`, `CategoryFilter`, `AutoRecompute`, `Port`. Five of
those are capability/scope, plus a string filter. Six things to think
about before the user has even decided what they want to do.

Concrete failure modes seen in practice:

- **Combinatorial confusion.** "Do I want `AllowScripting=True` +
  `ComponentScope=curated`? What does that even mean?" The answer
  depends on whether you're learning, teaching, or shipping —
  contexts the canvas doesn't know about.
- **Wrong default for the situation.** Students start with
  `AllowComponents=True, scope=defaults` and the AI starts placing
  things they don't recognize. Researchers start with the safer
  curated default and waste an hour fighting capability denials.
- **No "explain mode".** There's no way to tell the AI "just look at
  what's on the canvas and tell me what it does" without also
  authorizing it to mutate things.
- **Skill suggestions are advisory, not enforced.** A `landform`
  Skill *recommends* particular components but the engine doesn't
  hold the AI to that recipe — Skill becomes a hint, not a contract.

The root cause: the canvas axes are **what the engine permits**, not
**what the user is trying to do**. The redesign flips that.

## The five Scenarios

User picks ONE. The engine derives the flags.

| Scenario | "I want to…" | What AI does | Flag mapping (internal) |
|---|---|---|---|
| **🔍 Inspect** | "Explain this .gh file to me" | Reads canvas, summarises, answers questions. **Never** writes. | params=R, components=R, scripting=R |
| **🎛️ Tune** | "I have a working definition; just help me adjust parameters" | Reads + adjusts sliders/toggles/value-lists. **Cannot add or remove components.** | params=W, components=R, scripting=R |
| **🎓 Coach** | "Walk me through edits, and show me what you changed" | Tunes + may add components from the current Skill's allowed set. **Every change is visually highlighted on canvas and explained in chat.** | params=W, components=W (from Skill list), scripting=R |
| **📋 Execute** | "I have a Skill recipe; follow it exactly, no improvisation" | Follows the current Skill's command grammar. Cannot place anything outside the Skill's declared components. | varies — defined by Skill frontmatter |
| **🛠️ Author** | "Free hands — wire, place, script as needed" | Full authority. The current v0.1 default for advanced users. | params=W, components=W (scope=defaults\|all), scripting=W |

R = read, W = write.

Two design choices worth flagging:

- **Coach is a new behaviour, not a re-labelling.** Inspect/Tune/Author
  re-package existing capabilities. Coach needs new infrastructure
  (see §"Coach infrastructure" below).
- **Execute makes Skills binding.** Today Skills are advice. In
  Execute mode the Skill's `commands:` table in frontmatter defines
  the allowed user-facing intents and the AI is gated to them.

## Canvas surface change

Before (v0.1):

```
[ Run ] [ AllowParameters ] [ AllowComponents ] [ AllowScripting ]
[ ComponentScope ] [ CategoryFilter ] [ AutoRecompute ] [ Port ]
```

After (v0.2, default):

```
[ Run ] [ Scenario: dropdown ] [ Active Skill: dropdown ] [ Port ]
```

After (v0.2, "Advanced" expanded):

```
[ Run ] [ Scenario ] [ Active Skill ] [ Port ]
  └── Advanced
       [ Override AllowParameters ] [ Override AllowComponents ]
       [ Override AllowScripting ] [ Override ComponentScope ]
       [ AutoRecompute ]
```

The Advanced overrides exist for backward compatibility and for the
few research workflows that need a non-standard combination. They are
**opt-in**: default UI never shows them.

`CategoryFilter` is removed — categories are now declared by the
active Skill, not typed into a string.

## Skills schema v2

A Skill bundle is no longer just `SKILL.md` + `.ghuser`. It's a
versioned directory with five subfolders, each with a defined role.

```
skills/<skill-id>/
├── SKILL.md              # frontmatter + recipe
├── components/           # .ghuser canonical user-objects
├── reference/            # worked .gh examples (NEW)
├── images/               # sketches, photos, target outputs (NEW)
├── prompts/              # reusable prompt fragments per-mode (NEW)
└── tests/                # smoke-test scripts
```

### SKILL.md frontmatter

```yaml
---
name: facade-panelization
version: 0.2.0
description: One-line, used for AI selection
modes: [inspect, tune, coach, execute]   # which scenarios are supported
required_plugins: []
required_capabilities:
  scripting: false
component_scope: curated
allowed_categories: [Surface, Curve, MCP, Vector]
required_components:
  - { category: Surface, name: "Surface Box" }
  - { category: MCP, name: "Facade Panel" }
reference_examples:
  - { file: reference/basic.gh, when: "starting from scratch" }
  - { file: reference/parametric.gh, when: "user wants sliders for everything" }
  - { file: reference/with-glazing.gh, when: "user mentions windows or openings" }
images:
  parti: { file: images/parti-sketch.png, used_in: [inspect, coach] }
  precedent: { file: images/built-precedent.jpg, used_in: [inspect] }
commands:                       # ONLY for Execute mode
  panelize:
    description: "Generate a panelized facade from a surface"
    args: { surface: "Surface to panelize", divisions: "int" }
    steps: [...]
---
```

The frontmatter is the *contract*. The markdown body that follows is
the human-readable explanation.

### How reference files get used

Per scenario:

- **Inspect**: AI doesn't load reference files into the user's canvas.
  It may *read* them (via `gh_introspect_definition`) to ground its
  explanations of what the user already has.
- **Tune**: ignored (no additions allowed).
- **Coach**: AI matches user intent against `reference_examples[].when`
  descriptions, picks the closest, and loads it via
  `gh_load_skill_reference(skill_id, ref_name)`. Loading places the
  reference's components at a chosen pivot on the current canvas with
  visual highlighting (see Coach infrastructure).
- **Execute**: a command may declare a reference file as its starting
  point. The Skill author controls this in frontmatter, not the AI.

### How images get used

Each image declares `used_in: [modes]`. When the user is in one of
those modes and the Skill is active, the image is included in the
AI's context. Two channels:

- AI-side read (Claude vision): the bridge sends the image bytes to
  the model on every turn that touches this Skill.
- Client-side display (Claude Desktop / Cursor): the image is
  attached to the AI's reply so the user sees it.

The reference implementation will use client-side display first —
simpler, no per-turn token cost — and add AI-side read only when a
mode explicitly needs vision (e.g. a future "sketch-to-graph" Skill).

## Coach infrastructure (new)

Coach is the only mode that needs net-new functionality beyond
flag rearrangement.

### Turn-bounded change tracking

The bridge tags every component/wire it adds or modifies with the
ID of the current AI turn. When the turn ends, the tag becomes a
"highlight" marker that the canvas can render.

New bridge state:

```csharp
private static long currentTurnId = 0;
private static Dictionary<long, HashSet<Guid>> turnChanges = new();
```

New tools:

- `gh_begin_turn() -> turn_id` — called by the server at the start of
  each AI response.
- `gh_end_turn(turn_id)` — called when AI finishes. Triggers
  highlighting on canvas.
- `gh_dismiss_highlights(turn_id?)` — clears one turn's or all turn's
  highlights.

### Visual highlighting on canvas

Grasshopper components have a `Selected` boolean and component
attributes can override fill colour. Easiest implementation: set a
custom display mode that renders highlight-tagged components with a
colored badge (e.g. teal stroke for "added", orange for "modified").
Falls back to standard appearance after `gh_dismiss_highlights`.

Open question: do we render highlights as **canvas-side overlay**
(needs custom IGH_DrawHandler) or piggyback on `IGH_PreviewObject`?
TBD.

### "What I just did" summary

In Coach mode, the AI is prompted to always end its reply with a
structured summary:

```
What I changed in this turn:
- Added: Surface Box (turns lot polygon into solid)
- Added: Facade Panel (panelises the south face)
- Wired: Surface Box.B → Facade Panel.S
- Param tweak: floor_count 5 → 8
```

This is a prompt-side concern, not a code change — lives in
`skills/<skill-id>/prompts/coach.md` and gets injected as a system
message whenever Coach mode is active.

## Execute mode infrastructure

The AI is gated to the active Skill's `commands:` table. The Skill
author defines what user intents the AI can match against, the args
each accepts, and the tool sequence for each.

Implementation:

1. Skill `commands:` parsed at load.
2. AI's system prompt includes "your allowed commands are: [list]; if
   the user asks for anything else, respond 'this Skill doesn't
   support that — try Coach or Author mode'."
3. For each command, the `steps:` field is the explicit tool sequence
   — AI cannot deviate.

This is the RockStartIT pattern ([arXiv 2512.11882](https://arxiv.org/abs/2512.11882))
applied to parametric design: structured retrieval > hoping the model
follows instructions.

## Mapping table — the cheat sheet

| User-visible Scenario | Override panel default values |
|---|---|
| Inspect | params=False, components=False, scripting=False, scope=curated |
| Tune | params=True, components=False, scripting=False, scope=curated |
| Coach | params=True, components=True, scripting=False, scope=curated (from Skill) |
| Execute | (entirely Skill-defined) |
| Author | params=True, components=True, scripting=True, scope=defaults |

`scope=anything` is **not** exposed via any Scenario — opt-in only via
Advanced overrides.

## Backwards compatibility

- The capability flags are not removed, just hidden by default.
- Existing `.gh` files with the v0.1 canvas component continue to work
  if the user has installed v0.1 — the v0.2 component is a new GUID.
- Existing Skills (`landform`, `ladybug-environmental`) need a
  frontmatter add. Their current functionality maps to Coach mode in
  the new framing.
- Capability-flag bridge protocol (`get_capabilities`,
  `_parameterWriteCommands`, etc.) unchanged. Server only changes its
  derivation of `(params, components, scripting)` from the new
  `scenario` knob.

## Migration plan

1. v0.2.0-alpha: new canvas component + Scenario dropdown, mapped to
   existing capability flags. No Coach infrastructure yet, no Execute
   gating. Behaviourally same as v0.1.7 minus the messy UI.
2. v0.2.0-beta: Skills schema v2. Existing 2 Skills migrated.
3. v0.2.0-rc1: Coach infrastructure (turn tracking + highlighting).
4. v0.2.0-rc2: Execute mode gating.
5. v0.2.0: ship + deprecate v0.1 component.

Each alpha/beta/rc independently shippable.

## Open questions

- **Coach highlighting visual style.** Custom IGH_DrawHandler vs.
  abusing Selected/Display flags. Need a Grasshopper-API spike.
- **Reference file load — pivot placement.** When `gh_load_skill_reference`
  drops a definition into the canvas, where does it go? Cursor
  position? Auto-tile next to existing content?
- **Skill commands grammar — full grammar or sketch?** Are
  `commands:` enough, or do we need a state machine ("after panelize,
  user can call apply_glazing")?
- **Vision integration.** Does Coach mode pass reference images on
  every turn (token cost) or only when the user asks for visual
  comparison?
- **Where does Scenario state live?** Per-canvas (a `.gh` file
  remembers its mode), per-server (CLI flag), or both?
